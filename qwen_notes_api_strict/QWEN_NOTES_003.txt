#### Design Principles
1. **Offline-First**:
   - OpenJarvis is designed to operate entirely without network access.
   - Core functionalities like inference, memory, agents, tools, and telemetry are local by default.
   - Cloud APIs are optional extensions but never required for core operations.

2. **Hardware-Aware**:
   - Auto-detection of system hardware at startup (NVIDIA GPUs, AMD GPUs, Apple Silicon, CPU, RAM).
   - Recommendation of the optimal inference engine based on detected hardware.
   - Detection methods include `nvidia-smi`, `rocm-smi`, `system_profiler SPDisplaysDataType`, `/proc/cpuinfo`, and `/proc/meminfo`.

3. **Telemetry-Native**:
   - Telemetry is a first-class concern, with automatic recording of timing, token counts, energy usage, and cost.
   - Telemetry records are stored in a local SQLite database.
   - Telemetry collection is best-effort and does not block the query flow if it fails.

4. **Python-First**:
   - Clean Python API provided through the `Jarvis` class.
   - No framework lock-in; SDK is a standard Python package with dataclass-based types and no required web framework.
   - Use of type hints throughout the codebase.
   - Optional dependencies via extras (`openjarvis[server]`, `openjarvis[memory-colbert]`, etc.).

5. **OpenAI-Compatible**:
   - API server implements OpenAI chat completions API format.
   - Supported endpoints include `/v1/chat/completions`, `/v1/models`, and `/health`.
   - Requests and responses match the OpenAI API specification.
   - Streaming responses use Server-Sent Events (SSE) with `data: [DONE]` termination.

6. **Standalone**:
   - Requires no external services for core functionality.
   - Includes everything needed to run the system or uses standard system libraries.
   - Optional features that require additional packages include FAISS memory, ColBERT memory, and BM25 memory.
